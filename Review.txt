二、锁
1.
（1）死锁是指两个或多个进程在执行过程中，因争夺资源而造成的一种互相等待的现象，导致它们都无法继续执行
（2）可重入的理解
我们理解你可能需要一个更直观的解释。让我们用日常生活的类比来帮助你理解“可重入”的含义。
想象一个场景：
你是一个学生，你手里有一把钥匙（这把钥匙就是锁）。
这把钥匙可以打开你家里的一扇门（这个门就是被同步保护的代码块）。
现在，你进了家门（第一次拿到钥匙，进入第一个同步块）。进了家之后，你发现卧室的门也锁着，而且这把钥匙同样能打开卧室的门（卧室门就是第二个同步块）。如果你手里已经有这把钥匙了，你会因为“再次开卧室门”而被困在客厅吗？当然不会，因为你已经有钥匙了，可以直接开门进去。
如果钥匙不可重入，那会发生什么？你进了家之后，想进卧室，系统说“你不能用这把钥匙再开卧室门，你需要先放下钥匙，等别人还给你再开”，但你还没放下钥匙，怎么可能再获得？这就死锁了。
而可重入就是允许你“用同一把钥匙再次打开同一把锁保护的其他门”。在Java中，当一个线程已经获得某个对象的锁后，它再尝试获取同一个对象的锁时，会被允许，并且内部计数器会增加（比如记录你进了几次门），等退出时依次递减，直到计数器为0才真正释放锁。

2.ReentrantLock 是可重入锁
（1）同步代码块通常使用synchronized关键字来实现，它可以锁定一个对象或一个类，使得在对象或类的锁被占用时，其他线程无法进入被锁定的代码块。
ReentrantLock的lockInterruptibly()方法 可以响应线程中断，ReentrantLock.lockInterruptibly() 方法会以可中断的方式获取锁。如果调用该方法的线程在等待锁的过程中被其他线程中断，它不会一直阻塞，而是立即抛出 InterruptedException，从而允许线程响应中断（例如清理资源、退出任务等）。

4.乐观锁和悲观锁（无锁的两大派系）
悲观锁（Pessimistic Lock）是一种假设冲突会频繁发生的锁机制。每次数据访问时，都会先加锁，直到操作完成后才释放锁，这样可以确保在锁持有期间，其他线程无法访问这段数据，从而避免了并发冲突。
乐观锁：假设冲突较少，通常在数据提交更新时检测冲突，适用于读多写少的场景。实现方式一般是记录数据版本，若发现冲突则返回错误信息让用户决定如何处理。
公平锁：多个线程按照申请锁的顺序去获得锁，线程会直接进入队列去排队，永远都是队列的第一位才能得到锁。
非公平锁：多个线程去获取锁的时候，会直接去尝试获取，获取不到，再去进入等待队列，如果能获取到，就直接获取到锁。

5.cas（比较并交换）
cas的原理：
cas是一种无锁的并发控制机制，它通过比较内存中的值与预期值是否相等来判断是否需要更新内存中的值。
如果相等，则将预期值更新为新值；如果不相等，则不进行任何操作。
实际用途：
1. 电商/库存系统：防止超卖
这是最经典的场景。当多个用户同时抢购最后一件商品时，需要保证库存扣减的准确性。
传统方案：使用数据库行锁 select for update，性能较差，容易造成数据库连接堆积。
CAS 方案：利用数据库的乐观锁。
作用：如果执行结果返回的影响行数为 0，说明在读取库存到执行更新这段时间内，库存已经被别人修改了。业务层会捕获这个失败，进行重试或返回“库存不足”。
2. 金融/充值系统：账户余额更新
在扣款或充值场景，需要基于当前余额计算出新余额。
场景：用户 A 有 100 元，同时有两个请求进来：一个充值 50 元，一个消费 30 元。
作用：通过 CAS 思想（比较旧余额，设置新余额），避免了使用 select for update 锁住整行。只有第一个请求能成功更新，第二个请求会因为 balance 不再等于 old_balance 而失败重试。
3. 分布式系统：实现“只执行一次”的操作
在微服务或消息队列场景中，为了防止消息重复消费或重复执行定时任务，通常需要实现幂等。
场景：消息中间件（如 RocketMQ、Kafka）因网络抖动发送了重复消息；或者定时任务在多台服务器上同时触发。
CAS 实现：利用 Redis 的 setnx 命令或数据库唯一键/乐观锁。
流程：
尝试将任务 ID 写入 Redis，条件为“仅当 Key 不存在时”。
CAS 作用：setnx 成功，视为获得执行权，执行核心业务。
setnx 失败，说明其他机器正在执行或已执行过，直接丢弃任务。
进阶：使用 Redisson 等框架的可重入锁，其底层在 Redis 集群中也是通过 Lua 脚本封装了 CAS 逻辑来实现的。
4. 高并发计数场景
业务中常见的“今日访问量”、“点赞数”、“抽奖剩余次数”等，如果每次都去操作数据库，压力会很大。
场景：直播间观看人数实时更新。
CAS 实现：使用内存中的原子类（如 AtomicLong）或 Redis 的 INCR 命令。
作用：这些操作底层都是 CAS 实现的。多个线程同时对一个数值加 1，无需加锁，性能极高。最后再定时将内存中的累计值刷入数据库。
5. 业务状态机流转
在订单、工单等有明确状态流转的业务中，防止并发操作导致状态错乱。
场景：订单状态为“待支付”。用户发起了“支付”，同时客服发起了“取消订单”。
作用：只有当前状态符合预期（WAITING_PAY），才能更新为目标状态（PAID）。如果两个操作同时执行，只有一个会成功。另一个会因为影响行数为 0 而失败，业务层可以据此提示“订单状态已变更，请刷新”。
cas有个aba问题，a变成b再变成a，cas会认为没有变化，不会更新

7.
ArrayBlockingQueue
底层结构：基于数组的有界阻塞队列。
有界性：强制有界。必须在创建时指定容量，且容量一旦设置不可改变。
锁机制：一把锁。内部使用同一个 ReentrantLock 来控制入队（put）和出队（take）操作。这意味着生产者和消费者无法真正并行（不能同时进行插入和移除操作），但在低竞争场景下，锁开销更小且内存占用更稳定。
公平性：支持在构造函数中设置公平模式。公平模式下，等待时间最长的线程会优先获取锁，但这通常会降低吞吐量。
适用场景：适合容量固定且内存可控的场景，或者在需要保证公平性时使用。
LinkedBlockingQueue
底层结构：基于链表的阻塞队列。
有界性：可选有界。如果不指定容量，默认容量为 Integer.MAX_VALUE，即无界（实际受内存限制）。
锁机制：两把锁。内部维护了 takeLock（用于出队）和 putLock（用于入队）。在并发环境下，生产者和消费者可以同时操作队列头部和尾部，实现更高的吞吐量。
内存占用：由于是链表结构，每个元素需要额外的节点对象封装，GC 压力相对较大。
适用场景：适合高并发场景，通常比 ArrayBlockingQueue 拥有更高的吞吐量（尤其是在多线程竞争激烈时）。如果不指定容量，需注意防止生产者过快导致内存溢出。
作为大多数通用任务队列的首选
SynchronousQueue
底层结构：它不存储元素。可以理解为容量为 0 的队列。
核心机制：每个 put 操作必须等待一个 take 操作，反之亦然。它本质上是一个交接（Handoff）通道，生产者直接将数据交给消费者，数据不会在队列中停留。
实现模式：
公平模式：使用队列（FIFO）来存储等待线程，保证公平性。
非公平模式：使用栈（LIFO，后进先出）来存储等待线程，通常具有更好的吞吐量，但可能导致某些线程饥饿。
适用场景：常用于线程池（如 Executors.newCachedThreadPool），用于处理短时任务或传递性任务，因为不需要为数据保留存储空间，适合高吞吐量的请求-响应场景。

五、volatile与内存可见性（volatile 的作用）
问题背景：在多线程环境下，每个线程都有自己的工作内存（CPU 缓存）。线程对共享变量的修改可能只停留在工作内存，没有立即刷新到主内存，导致其他线程读取到旧值。
1. 内存可见性
写操作：当线程修改 volatile 变量时，JVM 会强制将该变量的值立即写回主内存，而不是只停留在缓存中。
读操作：当线程读取 volatile 变量时，JVM 会强制从主内存中读取最新值，而不是使用工作内存中的副本。
结果：一个线程对 volatile 变量的修改，对其他线程是立即可见的。
2. 禁止指令重排序
问题背景：为了提高性能，编译器和 CPU 可能会对指令进行重排序。但在多线程环境下，重排序可能导致意料之外的结果（例如双重检查锁失效）。
在 volatile 写操作前后插入内存屏障，防止该写操作之前的代码重排序到写操作之后，也防止之后的代码重排序到写操作之前。
在 volatile 读操作前后也插入屏障，确保读操作之后不会把后面的指令提前到读操作之前执行。
适用场景：
状态标记（boolean flag）
一写多读的场景
双重检查锁中的单例变量
原子性
使用 synchronized 同步代码块。
使用 AtomicInteger 等原子类（底层通过 CAS 实现）。
简单来说，volatile 用于更新，而 synchronized和AtomicInteger  用于读改写。
3.aqs（AbstractQueuedSynchronizer）是一个抽象同步框架（类），核心作用是统一分装了线程的等待和唤醒和排队机制，底层是通过一个volatile的变量state+fifo队列来实现了线程安全的资源抢夺，state表示资源的占用状态，0就是没有人站，1就是已上锁，线程抢不到锁就会被挂到队列里面排队，这个队列是变体的CLH队列，用双向列表实现，节点里还记录了线程的信息，等待状态等等，aqs不负责加不加锁，只是处理排队的细节，怎么加锁，怎么释放类得自己决定。重点（抽象框架、封装排队机制、核心是state+fifo的队列，真正的锁逻辑交给实现类自己定。

六、JVM是什么？
JVM（Java虚拟机）是运行Java字节码的虚拟计算机环境，它提供了一个平台无关的运行环境，使得Java程序能够在任何平台上运行而无需重新编写。
JVM五大内存区域名称及作用（程序计数器、虚拟机栈、本地方法栈、堆、方法区/元空间）
对象创建：类加载检查 → 分配内存（指针碰撞/空闲列表） → 初始化零值 → 设置对象头 → 执行（必背）
判断存活：可达性分析（GC Roots：虚拟机栈引用、静态属性引用、常量引用、JNI引用等）。（必背）
三大垃圾回收算法：1.标记-清除2.复制3.标记-整理（必背）

常考点（加分的深度）
Serial / Serial Old：单线程，简单，适合客户端模式。
Parallel Scavenge / Parallel Old：多线程，吞吐量优先，服务端默认。
CMS（Concurrent Mark Sweep）：并发收集，低停顿，但会产生碎片。
G1（Garbage First）：区域化，可预测停顿，Java 9+ 默认。

当一个类加载器收到加载请求时，先委派给父加载器，父加载器再向上委派，直到 Bootstrap 类加载器。只有父加载器无法加载时，子加载器才尝试自己加载。

OOM(内存溢出)
1.栈溢出 (StackOverflowError) 递归过深、线程栈帧过大 检查递归，或调小栈大小（-Xss）
2.堆内存溢出 (Java heap space) 对象过多、内存泄漏（如集合未清理）增大堆内存（-Xmx），或用MAT分析泄漏
3.方法区溢出 (Metaspace) 动态生成类过多（如大量代理类、JSP）
4.直接内存溢出 (Direct buffer memory) NIO中直接内存未释放 限制直接内存大小（-XX:MaxDirectMemorySize）

BIO / NIO / AIO 的区别（几乎必问）
   模型	特点	线程模型	适用场景

   NIO	同步非阻塞	一个线程管理多个Channel（Selector）	连接数多，短连接（聊天、游戏）

DispatcherServlet（中央控制器） 是 SpringMVC统一的入口，所有的请求都通过它，HandlerAdapter（处理器适配器）、HandlerMapping（处理器映射器）、ViewResolver（视图解析器）
DispatcherServlet 接收请求 → HandlerMapping 找到对应的 Controller 方法 → HandlerAdapter 执行该方法 → 返回 ModelAndView → ViewResolver 渲染视图 → 输出响应。建议结合SpringMVC的工作流程深度理解
4.Spring MVC
DispatcherServlet处理流程：接收请求 → 找HandlerMapping → 执行HandlerAdapter → 返回ModelAndView → 视图解析 → 渲染响应。
核心组件：HandlerMapping、HandlerAdapter、ViewResolver。

@EnableAutoConfiguration 注解用于启用 Spring Boot 的自动配置功能。它告诉 Spring Boot 应用程序自动配置应用程序所需的 Bean，从而减少了开发者的工作量，使得应用程序可以更快速地搭建和运行。
Spring Boot 启动时会扫描类路径下的 META-INF/spring.factories 文件，里面列出了所有自动配置类。每个自动配置类用 @Conditional 条件注解控制是否生效，比如 @ConditionalOnClass 表示只有类路径存在某个类时才生效。这样既实现了自动配置，又不会过度配置。

6.MyBatis就是把SQL与java对象通过映射绑定，并通过动态代理执行，流程分两步，启动时做初始化，运行时做执行，初始化阶段，项目启动时，Mybatis会先解析mybatis.config.xml，加载所有mapper文件，把数据源，插件、类型别名、数据映射全部注入到核心对象configuration，每一个sql优惠变成一个核心对象mapperstatement存进去，最后这些配置会被打包成一个SqlsessionFactory(mybatis的核心工厂类），然后到执行阶段，我们会先获取sqlsession，然后调用mapper方法，比如mapper.selectById(1),注意mapper接口没有实现类，mybatis会给他生成一个代理对象MapperProxy，当你调用方法时，MapperProxy它会去configuration里面找到对应的MapperStatement，交给Executor调度，StatementHandler准备SQL，ParameterHander绑定参数，ResultSetHandler映射结果，最后事务提交、回滚、关闭、链接，全部由SqlSession统一控制
